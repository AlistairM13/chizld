# Phase 3.1: Workout Splits - Research

**Researched:** 2026-03-09
**Domain:** SQLite schema design, React Navigation patterns, multi-zone exercise selection
**Confidence:** HIGH

## Summary

Phase 3.1 implements workout splits (templates) that allow users to create named workout presets combining exercises from multiple zones. This research covers the SQLite schema design for a many-to-many relationship between splits and exercises, navigation patterns for split management screens, and integration with the existing workout flow.

The core challenge is introducing a "split" entity that references exercises across zones while storing per-exercise default sets/reps. The existing codebase already has robust patterns for exercise selection (ExerciseSelectScreen with Set-based multi-select), database transactions (withExclusiveTransactionAsync), and navigation (native stack with typed params). The solution extends these patterns rather than introducing new paradigms.

**Primary recommendation:** Use a junction table `split_exercises` with `default_sets` and `default_reps` columns to store per-exercise settings within each split, enabling one split to reference exercises from any zone.

### Pattern 1: Junction Table for Many-to-Many
**What:** Splits reference exercises through a junction table that stores per-exercise defaults.
**When to use:** Any time a template needs to store configuration per item.
**Example:**
```typescript
// Source: Derived from existing schema patterns in database.ts
CREATE TABLE IF NOT EXISTS workout_splits (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  created_at TEXT DEFAULT (datetime('now')),
  updated_at TEXT DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS split_exercises (
  id TEXT PRIMARY KEY,
  split_id TEXT NOT NULL REFERENCES workout_splits(id) ON DELETE CASCADE,
  exercise_id TEXT NOT NULL REFERENCES exercises(id),
  default_sets INTEGER DEFAULT 3,
  default_reps INTEGER DEFAULT 10,
  sort_order INTEGER DEFAULT 0,
  created_at TEXT DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_split_exercises_split_id ON split_exercises(split_id);
```

### Anti-Patterns to Avoid
- **Storing exercise IDs as JSON array:** Loses referential integrity, can't query efficiently, harder to add per-exercise config.
- **Separate splits per zone:** Defeats the purpose of multi-zone workouts; zones should be optional filter, not mandatory grouping.
- **Re-using ExerciseSelectScreen directly:** That screen is zone-locked; split creation needs cross-zone selection with different navigation flow.

## Common Pitfalls

### Pitfall 1: Foreign Key Enforcement Not Enabled
**What goes wrong:** Deleting an exercise leaves orphan split_exercises rows.
**Why it happens:** SQLite foreign keys are OFF by default.
**How to avoid:** Already handled - database.ts sets `PRAGMA journal_mode = WAL` but should also set `PRAGMA foreign_keys = ON`. Use `ON DELETE CASCADE` on split_exercises.
**Warning signs:** Splits showing "Unknown Exercise" after deleting an exercise.

### Pitfall 2: Zone Filter Losing Selection State
**What goes wrong:** Changing zone filter clears the selected exercises.
**Why it happens:** FlatList re-renders with new data, Set<string> gets reset.
**How to avoid:** Keep selectedExercises state independent of the displayed list; filter is for display only.
**Warning signs:** User selects 3 biceps exercises, switches to traps, switches back, selections gone.

### Pitfall 3: Starting Workout Without Saving Split
**What goes wrong:** User creates a split inline, starts workout, split is lost.
**Why it happens:** Split was never persisted to database.
**How to avoid:** Two distinct flows: (1) Create/edit split screen that saves, (2) Start workout from saved split. Don't allow "Start" without "Save".
**Warning signs:** User taps "Start Workout" but split doesn't appear in list later.

### Pitfall 4: Sort Order Not Preserved
**What goes wrong:** Exercises appear in random order when loading split.
**Why it happens:** No ORDER BY in query, or sort_order not populated.
**How to avoid:** Use `sort_order` column in split_exercises, increment as exercises are added.
**Warning signs:** Exercises show in different order than user arranged them.

### useSplits Hook Structure
```typescript
// Source: Derived from useExercises.ts pattern
export function useSplits() {
  const db = useSQLiteContext();
  const [splits, setSplits] = useState<WorkoutSplit[]>([]);
  const [isLoading, setIsLoading] = useState(true);

  // Fetch all splits with exercise counts
  useEffect(() => {
    const fetchSplits = async () => {
      setIsLoading(true);
      const rows = await db.getAllAsync<SplitRow>(`
        SELECT ws.id, ws.name, ws.created_at,
               COUNT(se.id) as exercise_count
        FROM workout_splits ws
        LEFT JOIN split_exercises se ON se.split_id = ws.id
        GROUP BY ws.id
        ORDER BY ws.updated_at DESC
      `);
      setSplits(rows.map(/* transform */));
      setIsLoading(false);
    };
    fetchSplits();
  }, [db]);

  // Create split with exercises
  const createSplit = useCallback(async (
    name: string,
    exercises: { exerciseId: string; sets: number; reps: number }[]
  ) => {
    const splitId = generateId();
    await db.withExclusiveTransactionAsync(async (txn) => {
      await txn.runAsync(
        'INSERT INTO workout_splits (id, name) VALUES (?, ?)',
        [splitId, name]
      );
      for (let i = 0; i < exercises.length; i++) {
        const ex = exercises[i];
        await txn.runAsync(
          `INSERT INTO split_exercises
           (id, split_id, exercise_id, default_sets, default_reps, sort_order)
           VALUES (?, ?, ?, ?, ?, ?)`,
          [generateId(), splitId, ex.exerciseId, ex.sets, ex.reps, i]
        );
      }
    });
    refetch();
    return splitId;
  }, [db, refetch]);

  return { splits, isLoading, createSplit, updateSplit, deleteSplit };
}
```

### Starting Workout from Split
```typescript
// Source: Derived from ExerciseSelectScreen handleStart
const handleStartFromSplit = async (splitId: string) => {
  // Load exercises for this split
  const rows = await db.getAllAsync<SplitExerciseRow>(`
    SELECT se.exercise_id, e.primary_zone
    FROM split_exercises se
    JOIN exercises e ON e.id = se.exercise_id
    WHERE se.split_id = ?
    ORDER BY se.sort_order
  `, [splitId]);

  // Determine primary zone (most common, or first)
  const zoneCounts = rows.reduce((acc, r) => {
    acc[r.primary_zone] = (acc[r.primary_zone] || 0) + 1;
    return acc;
  }, {} as Record<string, number>);
  const primaryZone = Object.entries(zoneCounts)
    .sort((a, b) => b[1] - a[1])[0][0];

  const sessionId = generateId();
  navigation.navigate('WorkoutSession', {
    sessionId,
    exercises: rows.map(r => r.exercise_id),
    zoneId: primaryZone,  // For XP attribution
  });
};
```

## Open Questions

Things that require design decisions during planning:

1. **XP Attribution for Multi-Zone Splits**
   - What we know: Existing flow passes single `zoneId` to WorkoutSession
   - What's unclear: Should XP be split across zones or attributed to primary zone?
   - Recommendation: Use most-common zone as primary; splitting XP adds complexity for v1

2. **Train Tab Transformation**
   - What we know: Train tab currently shows placeholder "LOCKED" screen
   - What's unclear: Does user want splits ONLY, or splits + ad-hoc zone training?
   - Recommendation: Show split list with "Quick Train" button to access existing zone-based flow

3. **Default Sets/Reps Initialization**
   - What we know: Exercises table has no default_sets/default_reps columns
   - What's unclear: Should defaults come from exercise or be split-specific only?
   - Recommendation: Split-specific (3 sets, 10 reps as sensible default)

4. **Split Editing vs Versioning**
   - What we know: Splits can be edited
   - What's unclear: If user edits split mid-week, should workout history reference old or new version?
   - Recommendation: No versioning for v1; edits affect future workouts only
