# Phase 4.1: Workout Session Polish - Research

**Researched:** 2026-03-10
**Domain:** React Native UI layout, Reanimated animations, Gesture Handler slider
**Confidence:** HIGH

## Summary

This phase replaces the current `WorkoutSessionScreen` layout with a focused, minimal design. The current screen uses a scrolling list of completed sets, a multi-row controls bar, and stacked inputs — the new design collapses everything to a center block (exercise name + tempo countdown as dominant text), a single bottom row of inputs (Weight / Reps / RPE), corner indicators (elapsed time, running XP), and edge navigation arrows. The `RestTimerOverlay` gets simplified to a full black screen with only large countdown seconds, no progress ring.

The standard stack is already fully installed: Reanimated 4.2.1 for fade transitions and pulse animations, Gesture Handler 2.30.0 for the RPE horizontal slider, expo-haptics for tactile feedback, and Skia 2.5.1 (already used in RestTimerOverlay — this phase removes the Skia ring from rest timer, simplifying it).

The key implementation challenge is the two-state screen (active set view vs. rest view) with a crossfade transition. This is straightforward with Reanimated's `FadeIn`/`FadeOut` entering/exiting animations on conditional views, or with `withTiming` opacity on a shared value. A new `useElapsedTimer` hook is needed — the existing `useRestTimer` handles rest only; session elapsed time requires a separate interval counting up from session start.

**Primary recommendation:** Restructure `WorkoutSessionScreen` into two conditionally-rendered `Animated.View` blocks (active set view and rest view) with `FadeIn`/`FadeOut` layout animations. Build a custom RPE slider using `Gesture.Pan()` from Gesture Handler v2 (new API) clamped to the 6-10 range. Replace the current scrolled completed-sets list with a hidden-by-default summary count. Keep all existing hooks (`useWorkoutSession`, `useTempoVoice`, `useRestTimer`, `useXPService`) — they don't need changes.

### Pattern 1: Two-State Screen with Fade Transition

**What:** Conditionally render two `Animated.View` blocks using Reanimated `entering={FadeIn}` / `exiting={FadeOut}`. When a set is completed, hide the active view (FadeOut) and show the rest view (FadeIn). When rest ends, reverse.

**When to use:** Clean two-state UIs where old and new content don't coexist.

**Example:**
```typescript
// Source: https://docs.swmansion.com/react-native-reanimated/docs/layout-animations/entering-exiting-animations
import Animated, { FadeIn, FadeOut } from 'react-native-reanimated';

// Active set view
{!showRestTimer && (
  <Animated.View
    key="active-view"
    entering={FadeIn.duration(300)}
    exiting={FadeOut.duration(300)}
    style={styles.activeView}
  >
    {/* center block, inputs, bottom bar */}
  </Animated.View>
)}

// Rest timer view
{showRestTimer && (
  <Animated.View
    key="rest-view"
    entering={FadeIn.duration(300)}
    exiting={FadeOut.duration(300)}
    style={styles.restView}
  >
    {/* full black, countdown, skip */}
  </Animated.View>
)}
```

**Key requirement:** Both views must use a stable `key` prop when conditionally rendered inside a parent, so Reanimated tracks mount/unmount correctly.

### Pattern 2: Visual Pulse on Input Change

**What:** When Weight, Reps, or RPE changes, animate a brief scale pulse on the value display to confirm the change visually (accompanies haptic).

**When to use:** Any input that uses +/- steppers where the user taps repeatedly.

**Example:**
```typescript
// Source: https://docs.swmansion.com/react-native-reanimated/docs/fundamentals/applying-modifiers
import { useSharedValue, useAnimatedStyle, withSequence, withTiming } from 'react-native-reanimated';

const scale = useSharedValue(1);

const pulse = () => {
  scale.value = withSequence(
    withTiming(1.15, { duration: 80 }),
    withTiming(1.0, { duration: 120 })
  );
};

const animatedStyle = useAnimatedStyle(() => ({
  transform: [{ scale: scale.value }],
}));

// Call pulse() inside handleIncrement/handleDecrement alongside Haptics.selectionAsync()
```

### Pattern 3: RPE Horizontal Slider with Gesture Handler v2

**What:** A custom horizontal track that maps drag position to integer values 6-10. Uses `Gesture.Pan()` (v2 API, not legacy `PanGestureHandler`) with Reanimated shared values. Haptic fires on each integer snap.

**When to use:** The 6-10 RPE range is only 5 values across a wide landscape screen — a slider is more tactile and faster than 5 buttons.

**Example:**
```typescript
// Gesture Handler v2 API (react-native-gesture-handler ~2.30.0)
import { Gesture, GestureDetector } from 'react-native-gesture-handler';
import Animated, { useSharedValue, useAnimatedStyle, runOnJS } from 'react-native-reanimated';

const TRACK_WIDTH = 220; // landscape gives plenty of space
const RPE_MIN = 6;
const RPE_MAX = 10;
const RPE_RANGE = RPE_MAX - RPE_MIN; // 4 steps

const offsetX = useSharedValue(0); // 0..TRACK_WIDTH

const panGesture = Gesture.Pan()
  .onBegin((e) => {
    // capture start
  })
  .onChange((e) => {
    offsetX.value = Math.max(0, Math.min(TRACK_WIDTH, offsetX.value + e.changeX));
    const rpe = Math.round((offsetX.value / TRACK_WIDTH) * RPE_RANGE) + RPE_MIN;
    runOnJS(handleRPEChange)(rpe);
  });
```

**Note:** `GestureHandlerRootView` must wrap the app root or the screen — it's typically already in `App.tsx` for projects using Gesture Handler. Verify this is in place.

### Pattern 4: Elapsed Timer (Counting Up)

**What:** A hook that tracks seconds elapsed since session start. Uses the same drift-free absolute time pattern from `useRestTimer`.

**Example:**
```typescript
// New hook: useElapsedTimer.ts
// Same pattern as useRestTimer [04-02] but counting UP from startTime
export function useElapsedTimer() {
  const [elapsed, setElapsed] = useState(0);
  const startTimeRef = useRef<number>(0);
  const intervalRef = useRef<ReturnType<typeof setInterval> | null>(null);

  const start = useCallback(() => {
    startTimeRef.current = Date.now();
    intervalRef.current = setInterval(() => {
      setElapsed(Math.floor((Date.now() - startTimeRef.current) / 1000));
    }, 1000);
  }, []);

  const stop = useCallback(() => {
    if (intervalRef.current) clearInterval(intervalRef.current);
  }, []);

  // Format: "12:34" for MM:SS
  const formatted = `${Math.floor(elapsed / 60).toString().padStart(2, '0')}:${(elapsed % 60).toString().padStart(2, '0')}`;

  return { elapsed, formatted, start, stop };
}
```

### Anti-Patterns to Avoid

- **Animating from `StyleSheet.absoluteFillObject` with `entering/exiting`:** The overlay pattern (position absolute, fill parent) works fine with FadeIn/FadeOut — but `key` prop is mandatory when toggling between two conditional `Animated.View` children or React will reuse the same node and not trigger the animation.
- **ScrollView for a screen that should be static:** The current screen wraps content in a ScrollView. The new design is landscape-only and will not scroll — remove ScrollView entirely.
- **Legacy PanGestureHandler API:** The project has RNGN 2.30.0 — use `Gesture.Pan()` + `GestureDetector` (v2 API), not the legacy `PanGestureHandler` + `onGestureEvent` prop API. The v2 API is the current standard.
- **Removing `GestureHandlerRootView` wrapper:** If it's not already at App root level, the RPE slider pan gesture will silently fail. Check App.tsx.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| RPE slider drag | Custom touch responder | `Gesture.Pan()` from RNGH | Touch responders miss edge cases (touch cancel, fast swipe), RNGH handles them natively |
| Fade between views | Timed setState + opacity hack | Reanimated `FadeIn`/`FadeOut` entering/exiting | Entering/exiting runs on UI thread, avoids JS bridge jank |
| Elapsed timer | Browser `Date.now()` diff in render | Ref-based absolute time + setInterval (same as `useRestTimer`) | Prevents drift, proven pattern already in codebase |
| Visual pulse on value change | CSS-style transition | `withSequence(withTiming, withTiming)` on a shared value | Reanimated runs on UI thread, zero jitter |

**Key insight:** The RPE slider is the only genuinely new interaction pattern. Everything else is structural rearrangement of existing components.

## Common Pitfalls

### Pitfall 1: `entering`/`exiting` Not Firing on Toggle

**What goes wrong:** FadeIn/FadeOut doesn't animate — the view just appears/disappears instantly.

**Why it happens:** Two `Animated.View` blocks rendered conditionally without stable `key` props. React reconciles them as the same element and doesn't unmount/remount, so Reanimated never sees mount/unmount events.

**How to avoid:** Always use `key="active-view"` and `key="rest-view"` (or any stable unique keys) on the two state views.

**Warning signs:** Animation works first time but not on subsequent toggles.

### Pitfall 2: RPE Slider `GestureHandlerRootView` Missing

**What goes wrong:** Pan gesture on RPE slider does nothing — no error thrown.

**Why it happens:** `GestureDetector` requires `GestureHandlerRootView` to be an ancestor. Without it, gestures are silently dropped.

**How to avoid:** Check `src/App.tsx` — if `GestureHandlerRootView` isn't already wrapping the tree, add it. It wraps everything, not just the slider.

**Warning signs:** Tap gestures work (Pressable) but pan/drag gestures don't.

### Pitfall 3: Skia Canvas in Rest Timer Causes Stale Progress

**What goes wrong:** After simplifying the rest timer overlay (removing the Skia ring), the progress arc math still runs and creates warnings or stale calculations.

**Why it happens:** The current `RestTimerOverlay` computes `createArcPath(progress)` on every render. Removing Skia means removing the `Canvas` import and arc path logic entirely.

**How to avoid:** In the new rest timer design, delete all Skia imports and arc path code. The new design is pure RN views.

**Warning signs:** Import errors on `@shopify/react-native-skia` in the simplified component if the Canvas was removed but imports weren't.

### Pitfall 4: Tempo Countdown Display vs. Voice Tempo

**What goes wrong:** The UI shows "DOWN 3" visually but the voice says "Down 3" at a different time, creating a confusing mismatch.

**Why it happens:** `useTempoVoice` runs its own async loop independently. The visual "DOWN 3" counter in the center block would need to stay in sync with this loop — but `useTempoVoice` doesn't expose current phase/countdown state.

**How to avoid:** Either (a) extend `useTempoVoice` to expose `currentPhase` and `currentCount` state that the UI reads, OR (b) show a static "3-1-2-0" tempo label from the exercise config rather than a live countdown. Option (b) is simpler and avoids tight coupling. The context says "Tempo countdown is the visual focus when active" — clarify whether this means a live synced counter or a static display. Given implementation complexity, recommend the static approach unless user specifies otherwise.

**Warning signs:** Voice says "Down 2" while UI shows "DOWN 3" — they're independently timed.

### Pitfall 5: Corner Elements (Elapsed Time, XP Counter) Overlap Navigation Arrows

**What goes wrong:** In landscape layout, placing elapsed time and XP in corners while also having `< >` navigation arrows at screen edges creates overlapping touch targets.

**Why it happens:** Landscape screen is 800-900px wide but only ~400px tall. Corner elements at top-left/top-right and edge arrows at left/right center are in different vertical zones, but padding matters.

**How to avoid:** Position elapsed time + XP at top corners (top: 8, left/right: 12). Position `<` `>` arrows at left/right edges centered vertically in the active content area (not at screen edge corners). Use `position: absolute` for corner elements and arrows, keep them out of the flex flow.

### RPE Slider (Gesture Handler v2 + Reanimated)

```typescript
// Gesture Handler v2 — use Gesture.Pan() not legacy PanGestureHandler
import { Gesture, GestureDetector } from 'react-native-gesture-handler';
import Animated, { useSharedValue, useAnimatedStyle, runOnJS } from 'react-native-reanimated';

// TRACK_WIDTH should be measured with onLayout in production
const TRACK_WIDTH = 240;
const RPE_MIN = 6;
const RPE_MAX = 10;

function RPESlider({ value, onChange }) {
  const thumbX = useSharedValue(((value - RPE_MIN) / (RPE_MAX - RPE_MIN)) * TRACK_WIDTH);

  const pan = Gesture.Pan()
    .onChange((e) => {
      thumbX.value = Math.max(0, Math.min(TRACK_WIDTH, thumbX.value + e.changeX));
      const rpe = Math.round((thumbX.value / TRACK_WIDTH) * (RPE_MAX - RPE_MIN)) + RPE_MIN;
      runOnJS(onChange)(rpe);
    });

  const thumbStyle = useAnimatedStyle(() => ({
    transform: [{ translateX: thumbX.value }],
  }));

  return (
    <GestureDetector gesture={pan}>
      <View style={styles.track}>
        <Animated.View style={[styles.thumb, thumbStyle]} />
      </View>
    </GestureDetector>
  );
}
```

## Open Questions

1. **Tempo countdown: live-synced vs. static display**
   - What we know: `useTempoVoice` doesn't expose current phase/count. The context says tempo countdown should be a "visual focus when active."
   - What's unclear: Does "DOWN 3" mean a live counter that advances in sync with the voice, or a static display showing the exercise's configured tempo (e.g., "3-1-2-0")?
   - Recommendation: Implement as static tempo display (show configured values) unless user wants the complexity of syncing voice and UI. This can be extended later.

2. **GestureHandlerRootView presence in App.tsx**
   - What we know: RNGH is installed and used (Pressable gesture handling works), but `GestureDetector` for Pan requires `GestureHandlerRootView` as ancestor.
   - What's unclear: Whether `App.tsx` already wraps the root — needs verification before implementing RPESlider.
   - Recommendation: First task in phase should verify/add `GestureHandlerRootView` to `App.tsx`.

3. **Default sets target count**
   - What we know: The format "BENCH PRESS (2/5)" implies a target set count. Currently `WorkoutSession` doesn't receive a target set count per exercise — only an exercise ID list.
   - What's unclear: Where does "5" (total planned sets) come from? Split exercises have `defaultSets` but this isn't currently passed to the session.
   - Recommendation: Pass `defaultSets` per exercise in the session route params or derive from `tempoDefaults`. The `(X/Y)` display can fall back to showing only completed sets count if target is unknown.
